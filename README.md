## Mural // Nick - Marketplace backend

This project is hosted by Vercel and is reachable at the base url https://mural-jade.vercel.app. There are a handful of rest api endpoints exposed that allow users to place orders and for merchants to track customers' payment and conversion to a Colombian bank account.

### Interacting with the service

- From your terminal, set `BASE=https://mural-jade.vercel.app`
- Ensure the app is online by hitting the health endpoint `curl -s $BASE/health`, the expected response is `{"status":"ok"}`.
- I created an account in Mural that I called "Accounts Receivable" to act as the receiver for all payments. The address is `0x61ef58e23439B861B6dB5085796ff726f6B521F7` and should match the `depositWallet` when you place an order

- The following is a run through of the happy path for checking out, paying, and converting

```bash
# 1. Browse the product catalog, there should be 3 widgets priced 1, 2, and 3 USDC respectively
curl -s $BASE/products | jq

# 2. Place an order — note the payment.depositWallet and payment.amountUsdc in the response
curl -s -X POST $BASE/orders \
  -H "Content-Type: application/json" \
  -H "x-customer-id: alice" \
  -d '{"products": [{"productId": 3, "quantity": 1}]}' | jq

# 2a. View order status
curl -s $BASE/orders/{order_id}

# 3. Send exactly payment.amountUsdc USDC to payment.depositWallet on Polygon Amoy via MuralPay

# 4. Trigger the payment detection job
curl -s $BASE/cron/check-payments -H "Authorization: Bearer vercel-cron-secret" | jq

# 5. Confirm the order is paid and COP conversion was initiated
curl -s $BASE/merchant/orders -H "x-api-key: merchant-api-key" | jq

# 6. Check the COP withdrawal status (refreshed live from MuralPay on each call)
curl -s $BASE/merchant/payouts -H "x-api-key: merchant-api-key" | jq
```

### Current status

The happy path for all requirements are working.

For customer payment detection, I indended to have a cron job that would run every minute. Vercel's hobby project limits cron jobs to once per day, so we must call the endpoint to initiate "detection" of a new payment. The curl command for this is in the previous section.

For the recipient Colombian COP account, I created an External Account in the Mural UI. I indended to use this, but instead went with an "inline fiat object" for the payout. This lives in `/src/lib/mural.ts`. The Mural concepts of an External Account vs Inline Account vs Counterparty were confusing to me, and to complete this project I just went with the first method I could get working.

### Future work

The top areas to address for production readiness are:

1. Use webhooks for payment detection instead of polling. I chose a cron job because I am more familiar with this paradigm, but as a payment service we should be taking action as soon as the customer's payment has come through.
2. Implement authentication for both customers and merchants. This would allow us to create a better interface for customers by maintaining a cart for them across visits to the app. For merchants, we want to ensure their order data and payment conversion data is private to them only.
3. Address pitfalls to payment detection system (and support other methods of payment!)

- Order matching for same-priced orders
  Currently we naively match payments to orders based on the USDC amount. One customer could order 3 Basic Widgets while another orders 1 Deluxe Widget. The first payment of 3 USDC will match the first order returned from our `prisma.order.findMany()` invocation rather than the proper order. To combat this, we could assign unique amounts to send such as 3.000001 and 3.000009 USDC. We could also use a set of recipient addresses such that customers would interact with a unique address that we can tie to a particular order.
- Expire orders
  Orders that never expire are an issue because if a customer never pays, but a future customer creates an order for that same amount, the payment matching code could match the first order!
- Reconcile payment amounts - partial payments with wiggle room
  This is for way in the future, but we could allow customers to pay in multiple payments. We could also introduce a small tolerance (ex. 0.001 USDC) to account for rounding differences across wallets, without breaking the matching logic.
